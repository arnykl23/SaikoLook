3. スキーマ層 — models.py

class EmailMessage(BaseModel):
    id: str
    provider: str = "gmail"
    subject: str = ""
    sender: str = ""
    to: list[str] = []
    received_at: datetime | None = None
    snippet: str = ""
    is_unread: bool = False
  • pydantic の BaseModel。プロジェクト全体の「メール1通」の定義。
  • 役割① 共通の形: Gmail/Outlook どちらもこの形に揃える → 上位層は取得元を意識しない。
  • 役割② 変換: FastAPI がこれを使い datetime を ISO 文字列 JSON に自動変換。
  • 将来 body_text / importance を足す予定地（LLMフェーズ）。

4. 取得層 — fetch_gmail_imap.py

役割ごとに関数を分割。中核は fetch_emails()。

① 認証情報の取り出し

address = os.environ.get("GMAIL_ADDRESS", "").strip()
app_password = os.environ.get("GMAIL_APP_PASSWORD", "").replace(" ", "")

環境変数から読み、アプリパスワードの空白を除去。未設定なら RuntimeError。

② 接続

imap = imaplib.IMAP4_SSL("imap.gmail.com", 993)
imap.login(address, app_password)
imap.select("INBOX", readonly=True)   # 覗くだけ・既読化しない

③ 取得本体

status, data = imap.search(None, "ALL")
ids = data[0].split()
latest = ids[-limit:][::-1]            # 末尾=最新 を新しい順に
for num in latest:
    imap.fetch(num, f"(FLAGS BODY.PEEK[]<0.{PREVIEW_BYTES}>)")
ポイント意味
BODY.PEEK[]<0.16384>全文ではなく先頭16KBだけ取得。ヘッダは先頭にあるので件名等は完全に取れ、添付込み巨大メールでも軽量（レビュー指摘対応）。
FLAGS同時取得し、応答に \Seen が無ければ未読 → is_unread。
PEEKサーバ上の既読フラグを変えない。
finally: logout()必ず接続を閉じる。

④ 整形ヘルパー

  • decode() — =?UTF-8?B?...?= 形式の件名/差出人を人間可読に（日本語の文字化け防止）。
  • extract_snippet() — multipart を walk() して text/plain の先頭120字を抽出。

⑤ CLI 兼用

def main():
    for m in fetch_emails():
        print(...)   # 未読●付きで一覧表示

if __name__ == "__main__":
    main()           # python -m app.fetch_gmail_imap で実行

API と CLI が同じ fetch_emails() を共用しているのが設計上の要点。

5. API層 — main.py

app = FastAPI(title="ReplyGuard API", version="0.1.0")
app.add_middleware(CORSMiddleware, allow_origins=["*"], ...)

@app.get("/emails", response_model=list[EmailMessage])
def get_emails(limit: int = Query(MAX_RESULTS, ge=1, le=50)):
    try:
        return fetch_emails(limit=limit)
    except RuntimeError as e:
        raise HTTPException(status_code=500, detail=str(e))
  • 薄い層: HTTP の入口だけ。実処理は取得層に委譲。
  • response_model で出力型を保証し、自動 JSON 化 ＋ Swagger 生成。
  • limit は ge=1, le=50 で範囲バリデーション。
  • 認証エラー(RuntimeError)を HTTP 500 に変換。CLI 側は同じ例外を SystemExit で表示 → 層ごとに適切なエラー処理。
  • /health は起動待ち・死活監視用。

8. 設計上の効いている判断

  • 取得ロジックを fetch_emails() に1本化 → API も CLI も同じ実装を共用。
  • EmailMessage で出力形を固定 → 次の LLM 層は「結果にスコアを足す」だけ、UI 層は「/emails を叩く」だけで済む。
  • 層ごとのエラー変換（CLI=SystemExit / API=HTTP 500）で、どの入口でも適切に失敗を伝える。
