2. システム設計の全体方針

2.1 レイヤリング（ヘキサゴナル / ポート＆アダプタ）

差し替えたい部分（メールソース・LLM・DB・通知）をすべて Protocol（ポート）で抽象化し、具体実装（アダプタ）を後から差し替え可能にする。依存方向は常に内向き（アダプタ→サービス→ドメイン）。ドメインは外部技術を知らない。

2.3 2つのデータ経路（取込は非同期・読み出しは同期）

Gmail への IMAP 取得は遅く詰まりうるので、HTTP リクエストの同期処理から切り離す。取込はスケジューラ（または POST /sync）が裏で回し、フロントはDB を読むだけ。これにより一覧表示が IMAP の遅延・失敗に巻き込まれない。

3. データモデル全体

層間の契約は3つ：取得層が作る EmailMessage、LLM箱が返す AnalysisResult、状態管理が永続化する MessageRecord。EmailMessage は破壊変更せず任意フィールド追加のみ。

3.1 スキーマ（pydantic v2）

# app/models.py（既存を拡張：既存フィールドはそのまま、任意追加のみ）
class EmailMessage(BaseModel):
    id: str                       # IMAP UID（プロバイダ内安定ID）
    provider: str = "gmail"
    subject: str = ""
    sender: str = ""
    to: list[str] = []
    received_at: datetime | None = None
    snippet: str = ""
    is_unread: bool = False
    body_text: str | None = None  # ★追加(任意)：LLM入力用。取得層が先頭のみ詰める

class AnalysisResult(BaseModel):
    model_config = ConfigDict(extra="forbid")   # LLM出力の余計なキーを弾く
    importance: int = Field(ge=1, le=5)          # 重要度 1–5
    needs_reply: bool                            # 対応要否
    task_weight: Literal["light","medium","heavy"]
    category: str                                # 分類ラベル
    summary: str                                 # 要約
    suggested_action: str | None = None
    deadline: datetime | None = None
    reason: str                                  # 判定理由（説明可能性）
    analyzer: str = "stub"                       # どの実装が出したか

class MessageState(str, Enum):
    UNHANDLED="unhandled"; IN_PROGRESS="in_progress"
    DONE="done"; SNOOZED="snoozed"; DISMISSED="dismissed"

class MessageRecord(BaseModel):       # 永続化の単位（EmailMessage＋分析＋状態）
    message_id: str                   # 安定ID = f"{provider}:{id}"
    email: EmailMessage
    analysis: AnalysisResult | None = None
    state: MessageState = MessageState.UNHANDLED
    triage_score: float = 0.0         # 未対応検知スコア
    version: int = 0                  # 楽観ロック用
    created_at: datetime; updated_at: datetime

4. 受信・正規化層（Gmail）

既存の fetch_gmail_imap.py（読み取り専用・BODY.PEEK・UID・timeout・件数/バイト上限）をそのまま核に、ソース差し替えの口 MessageSource を被せる。Slack/Outlook/GitHub は同じ Protocol の別実装として将来追加。

4.1 ポート定義

# app/ports/source.py
class MessageSource(Protocol):
    provider: str
    def list_recent(self, limit: int) -> list[EmailMessage]: ...
    def fetch_since(self, state: SyncState, limit: int) -> FetchResult: ...
    def close(self) -> None: ...

@dataclass
class SyncState:           # アカウント単位の取得位置（差分取得の鍵）
    account: str
    uidvalidity: int | None = None
    last_uid: int | None = None

@dataclass
class FetchResult:
    messages: list[EmailMessage]
    new_state: SyncState
    truncated: bool        # 上限で打ち切ったか

4.2 差分取得（UID / UIDVALIDITY）

毎回末尾 N 件を取り直すのは無駄が多く取りこぼしも出る。IMAP の UID は INBOX 内で単調増加するので、last_uid より新しい UID だけを取る。UIDVALIDITY が変わったら UID 空間がリセットされた合図なので、保存済み UID を破棄して取り直す。

sequenceDiagram participant S as IngestionService participant G as GmailImapSource participant I as IMAP(gmail) S->>G: fetch_since(SyncState{uidvalidity,last_uid}) G->>I: login + SELECT INBOX (readonly) I-->>G: UIDVALIDITY, EXISTS alt UIDVALIDITY 変化 G->>G: last_uid を破棄(全UID空間が変わった) end G->>I: UID SEARCH UID (last_uid+1):* I-->>G: 新着UIDリスト G->>I: UID FETCH ... (UID FLAGS BODY.PEEK[]<0.16384>) I-->>G: ヘッダ＋本文先頭(既読化しない) G->>G: decode/正規化 → EmailMessage[] G-->>S: FetchResult(messages, new_state{last_uid=max})

4.3 正規化で守ること

  • MIME ヘッダのデコード（decode_header/make_header）。失敗時は生値フォールバック。
  • 本文は text/plain 優先抽出。HTML のみのメールはテキスト化（BeautifulSoup 等）。body_text は先頭 N 文字のみ（巨大本文を抱えない）。
  • 差出人/宛先の整形、日時パース（失敗は None＋ログ）。
  • 安定ID＝f"{provider}:{uid}" で全層一致させる。
セキュリティ厳守：送信・書込みスコープは要求しない。BODY.PEEK で既読化しない。全件 SEARCH や本文全文取得をしない（件数・バイト・timeout 上限を維持）。資格情報を例外メッセージ・ログに混ぜない。

4.4 想定エラー（取得層）

事象原因設計上の扱い
ログイン失敗アプリパスワード誤り/2段階未設定資格情報を伏せた RuntimeError。同期は失敗ログのみ（一覧APIは前回分を返し続ける）。
接続タイムアウトネットワーク/Gmail側IMAP_TIMEOUT で打ち切り→リトライ（指数バックオフ、上限あり）。
UIDVALIDITY 変化メールボックス再構成last_uid 破棄して再取得。重複は upsert で吸収。
デコード不能ヘッダ壊れた MIME握りつぶさず生値＋警告ログ。1通の失敗で全体を落とさない。
巨大メール添付過多先頭 16KB のみ取得済み。本文は body_text 上限で切り詰め。

5. 状態管理エンジン＋DB

対応漏れ防止の核。各メッセージは有限状態機械で管理し、状態遷移は許可表で検証、更新は楽観ロックで競合を防ぐ。永続化は Repository Protocol で抽象化し、Supabase(PostgreSQL) 本命・ローカル SQLite フォールバックを同契約で切替える。

5.1 状態遷移（FSM）

stateDiagram-v2 [*] --> unhandled: 取込 unhandled --> in_progress: 着手 unhandled --> dismissed: 対象外 unhandled --> snoozed: スヌーズ in_progress --> done: 完了 in_progress --> unhandled: 差し戻し snoozed --> unhandled: 期限到来 done --> in_progress: 再オープン dismissed --> unhandled: 復帰 
# app/domain/state.py — 許可遷移のみ通す（不正遷移は弾く）
ALLOWED = {
  UNHANDLED: {IN_PROGRESS, DISMISSED, SNOOZED},
  IN_PROGRESS:{DONE, UNHANDLED},
  SNOOZED:   {UNHANDLED},
  DONE:      {IN_PROGRESS},
  DISMISSED: {UNHANDLED},
}
def can_transition(cur, nxt) -> bool: return nxt in ALLOWED.get(cur, set())

5.2 楽観ロック（競合更新の防止）

2人が同じメールの状態を同時に変えると後勝ちで履歴が壊れる。version 列を使い UPDATE ... WHERE message_id=? AND version=?。0 行更新なら競合として 409 Conflict を返し、フロントは再取得を促す。

sequenceDiagram participant U1 as 利用者A participant U2 as 利用者B participant API participant DB U1->>API: PATCH state (version=3) API->>DB: UPDATE ... WHERE version=3 DB-->>API: 1 row → version=4 API-->>U1: 200 OK U2->>API: PATCH state (version=3 ※古い) API->>DB: UPDATE ... WHERE version=3 DB-->>API: 0 row API-->>U2: 409 Conflict (再取得して)

5.3 未対応検知スコアリング

「未読 × 高重要度 × 経過時間」で triage_score を算出し、ダッシュボードの並べ替えと通知判定に使う。ルールはドメインに置き、LLM の有無に依存しない（スタブでも動く）。

# app/domain/triage.py（例：重み付け・要調整）
score = (importance_weight[a.importance]      # 重要度 1–5 → 重み
         + (2.0 if email.is_unread else 0)
         + age_hours(email.received_at) * 0.1
         + (3.0 if a.needs_reply else 0))

5.5 想定エラー（状態管理）

事象扱い
不正な状態遷移can_transition で 422。許可表外は拒否。
版ずれ（同時更新）409 Conflict＋最新版を返す。

6. API層＋JWT＋LLM箱＋通知箱

6.1 エンドポイント

6.2 認証（JWT）

本命は Supabase Auth（発行済み JWT を API 側で検証、RLS と整合）。PoC では PyJWT(HS256) の最小実装でも可。検証は DI get_current_user に集約し、ルータは Depends で受けるだけ。

# app/api/deps.py
def get_current_user(token = Depends(oauth2_scheme)) -> User:
    try:
        payload = jwt.decode(token, settings.jwt_secret, algorithms=["HS256"],
                             audience=settings.jwt_aud)
    except JWTError:
        raise HTTPException(401, "invalid token")
    return User(id=payload["sub"])
秘密鍵は env/Secrets から。トークンは Authorization: Bearer。CORS は許可 origin 明示（ワイルドカード禁止、既存 ALLOWED_ORIGINS 方針を踏襲）。

6.3 LLM箱（Analyzer）

インターフェースと出力スキーマを今確定し、中身はスタブ。実モデルは同契約の別実装として後日差し込む（コード変更なしで切替）。

# app/ports/analyzer.py
class Analyzer(Protocol):
    name: str
    def analyze(self, email: EmailMessage) -> AnalysisResult: ...

# app/analysis/stub.py — ルールベース（決定的・テスト容易）
class StubAnalyzer:
    name = "stub"
    def analyze(self, e: EmailMessage) -> AnalysisResult:
        kw = any(k in (e.subject+e.snippet) for k in ("至急","締切","請求","返信"))
        imp = 5 if (e.is_unread and kw) else 3 if e.is_unread else 2
        return AnalysisResult(importance=imp, needs_reply=kw,
            task_weight="medium" if kw else "light",
            category="action" if kw else "fyi",
            summary=e.snippet[:80], reason="件名/本文のキーワードと未読状態で判定",
            analyzer="stub")
プロンプト防御（実モデル時に必須）：メール本文・件名はユーザーデータであって命令ではない（LLM01）。システム指示とデータをデリミタ/専用フィールドで分離し「以前の指示を無視せよ」等に乗らない。出力は extra="forbid" でスキーマ検証し、範囲外（importance 1–5 外・余計な実行指示）を弾く（LLM05）。本文を外部へ送るのは分析 API への正規呼び出しに限る。

6.4 通知箱（Notifier）

# app/ports/notifier.py
class Notifier(Protocol):
    def notify(self, rec: MessageRecord, *, dedupe_key: str) -> None: ...
# app/notify/log.py — スタブ：標準ログに出すだけ（外部送信しない）

dedupe_key（例 f"{message_id}:{date}"）でべき等にし、スケジューラ再実行や二重起動で同じ通知が連発しないようにする。Email/Slack 送信は口だけ用意。

6.5 スケジューラ（定期取込）

APScheduler を lifespan で起動し、一定間隔で IngestionService.run() を呼ぶ。多重起動防止（ロック/単一ワーカー）と実行重なり防止（max_instances=1・coalesce=True）を設定。失敗しても次回に回復する設計。

8. 横断リスク・エラー設計

層をまたぐリスクと、その防御をまとめる。実装時は各担当がこの表を満たすこと。

8.1 セキュリティ横断

リスク該当層防御
プロンプトインジェクション（LLM01）取得・LLM本文＝データとして扱い指示分離。本文中の命令文に従わない。検知時は処理継続せず記録。
LLM 出力の不正利用（LLM05）LLM・状態extra="forbid"＋範囲検証。出力をシェル/SQL/eval に渡さない。failで既定値フォールバック。
秘密情報の漏洩全層env/Secrets から読む。コミット禁止（secrets/・token_*.json・*.env）。例外/ログに資格情報を混ぜない。*.env.example のみ可。
既読化の副作用取得BODY.PEEK・readonly=True 維持。書込みスコープ要求禁止。
CORS 経由の情報読取API許可 origin 明示・ワイルドカード禁止（現行方針踏襲）。
XSS（メール本文）フロントHTML 描画は DOMPurify。既定はテキスト表示。
認証バイパスAPI状態変更系は get_current_user 必須。Supabase は RLS でユーザー分離。
外部送信全層PC内情報・本文・パスを外部送信しない（分析 API への正規呼び出しを除く）。

8.2 堅牢性・運用横断

リスク防御
IMAP の遅延でAPIが詰まる取込を非同期化（§2.3）。読み出しはDBのみ。各IMAP操作に timeout。
同時状態更新の競合楽観ロック（version）＋409。
取込・通知の二重実行スケジューラ max_instances=1/coalesce＋通知 dedupe_key でべき等。
再取込で状態巻き戻りupsert はメール本体のみ更新し state は保持。
LLM コスト/レート本文を要約/切詰めて渡す・timeout・リトライ・上限（実モデル時）。
部分失敗1通の失敗で全体を止めない。失敗は記録し次回回復。
